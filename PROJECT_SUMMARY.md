# Qwen3-4B 单文件 Python 诊断微调项目总结

## 1. 项目背景

本项目围绕一个范围明确的窄任务展开：给定用户输入的单文件 Python 代码片段、报错信息或“结果不对”的描述，模型输出结构化的诊断结论、最小修复建议、修复后的代码片段、风险提示以及固定 JSON 摘要。

相较于通用代码助手，这个任务更强调三件事：

- 诊断结论尽量准确，尤其是 `error_type` 与 `root_cause`
- 修复方案尽量保持最小改动，而不是大段重写
- 面对信息不足或报错与代码不匹配的场景时，要优先追问或拒绝臆造

项目的基础模型为 `Qwen/Qwen3-4B-Instruct-2507`，训练路线采用两阶段方案：

1. LoRA SFT
2. LoRA DPO


## 2. 任务定义

### 2.1 输入边界

当前版本只处理单文件 Python 片段，输入主要覆盖以下几类场景：

- traceback + 代码片段
- 无异常但结果不符合预期
- 报错信息与代码片段不匹配
- 信息不足，需要模型主动追问

### 2.2 目标输出

模型需要按照统一结构回答，并在末尾输出固定 JSON 摘要。目标字段如下：

```json
{
  "error_type": "string",
  "root_cause": "string",
  "fix_summary": "string",
  "risk_note": "string",
  "need_more_context": false,
  "confidence": "low|medium|high"
}
```


## 3. 数据与训练方案

### 3.1 数据构成

当前项目中的训练数据来自“教师模型蒸馏 + 人工筛选 + 偏好构造”三部分组合：

- 高质量人工种子样本
- 围绕种子样本扩展出的监督数据
- 为 DPO 构造的 `chosen / rejected` 偏好对

当前正式训练集规模如下：

- SFT：`1824` 条
- DPO：`1824` 条

主要覆盖的错误类型包括：

- `type_error`
- `value_error`
- `index_error`
- `key_error`
- `attribute_error`
- `module_not_found`
- `import_error`
- `syntax_error`
- `indentation_error`
- `zero_division`
- `logic_error`
- `fstring_error`
- `missing_context`
- `mismatched_context`

### 3.2 训练流程

训练采用标准的两阶段路线：

1. 先做 SFT，让模型学会结构化回答格式、JSON 输出习惯以及任务边界
2. 再做 DPO，强化“最小修复、少幻觉、信息不足先追问”等偏好行为

两轮训练均已完整跑通，训练曲线整体平稳，没有出现显著的数值异常、梯度爆炸或训练中断问题。


## 4. 评测设计

### 4.1 benchmark 组成

项目当前保留了一份 `20` 条的 benchmark 种子集，并拆分为：

- `benchmark_dev_v1.json`：`8` 条
- `benchmark_test_v1.json`：`12` 条

此外，也可以直接使用完整集：

- `data/train_data/data_v1.json`：`20` 条

### 4.2 当前评测指标

离线评测脚本会统计以下指标：

- 样本覆盖率
- 严格关键点命中率
- 宽松关键点命中率
- `error_type` 准确率
- JSON 合法率
- `need_more_context` 准确率

其中需要特别说明的是：

- 严格关键点命中率采用原句包含判断，当前基本可以视作参考指标，不适合作为主指标
- 宽松关键点命中率是基于规则的概念匹配，用于近似判断模型是否回答到了关键语义

### 4.3 评测局限

当前评测仍然偏文本启发式，尚未覆盖下面两个更强的维度：

- 根因判断的人工主观评分
- 修复代码是否可执行、是否通过断言

因此，当前结论可以用于阶段性比较，但还不能完全替代更严格的可执行评测。


## 5. 实验结果

下面给出在完整 `20` 条 benchmark 上的三组模型对比结果。

| 模型 | 宽松关键点命中率 | error_type 准确率 | JSON 合法率 | need_more_context 准确率 |
|------|------------------|------------------|-------------|--------------------------|
| Base | 70.18% | 75.00% | 90.00% | 90.00% |
| SFT  | 57.89% | 85.00% | 100.00% | 95.00% |
| DPO  | 56.14% | 90.00% | 95.00% | 95.00% |


## 6. 结果分析

### 6.1 Base 模型的特点

原始 `Qwen/Qwen3-4B-Instruct-2507` 在通用 Python 报错分析上已经具备较强能力，尤其是在常见、直接的异常场景中，自然解释能力较强，因此宽松关键点命中率最高。

但 Base 模型也暴露出几个明显短板：

- 输出格式不够稳定
- `error_type` 标注不够一致
- 面对信息不足场景时，追问行为不够可靠

换句话说，Base 更像“会说”，但不够“受控”。

### 6.2 SFT 模型的特点

SFT 阶段的收益最明确，主要体现在任务约束对齐上：

- `JSON 合法率` 提升到 `100%`
- `error_type` 准确率从 `75%` 提升到 `85%`
- `need_more_context` 准确率从 `90%` 提升到 `95%`

这说明 SFT 成功地把模型从“通用代码助手”收束成了更接近目标任务的“结构化诊断器”。

与此同时，SFT 也带来一个可见代价：

- 某些题上的自然解释丰富度下降
- 部分逻辑错误题的宽松关键点命中率相较 Base 有回退

因此，SFT 的主要贡献不是让模型第一次学会诊断 Python 错误，而是让模型更稳定地按规定方式输出。

### 6.3 DPO 模型的特点

DPO 进一步把 `error_type` 准确率提高到了 `90%`，说明偏好对齐在“标签一致性”和部分行为约束上是有效的。

但从整体结果看，DPO 还没有形成对 SFT 的全面优势：

- `JSON 合法率` 从 `100%` 回落到 `95%`
- 宽松关键点命中率没有超过 SFT
- 个别简单样本上出现局部回退

因此，当前 DPO 更适合被理解为“方向正确，但还需要继续打磨”的增强版本，而不是已经稳定超越 SFT 的最终交付版。


## 7. 当前阶段结论

基于现有训练结果与完整 benchmark 对比，可以得到以下阶段性结论：

1. 原始基座模型在通用 Python 报错诊断上已经具备较强能力，因此本项目的价值不在于“让模型第一次学会报错分析”，而在于任务约束对齐。
2. SFT 显著提升了结构化输出稳定性、错误类型一致性和信息不足场景下的追问行为，是当前最均衡、最适合作为第一版交付模型的方案。
3. DPO 在错误类型一致性上进一步带来收益，但目前仍存在局部样本回退，说明偏好数据还需要针对性优化。

因此，当前版本中最稳妥的工程交付候选为：

- **SFT 模型**

而 DPO 版本的定位更适合写成：

- **具备潜力的下一阶段优化方向**


## 8. 验收标准建议

如果将本项目继续推进到更严肃的实验或产品化阶段，建议使用下面这组验收标准：

- `JSON 合法率 >= 98%`
- `error_type` 准确率 >= `85%`
- `need_more_context` 准确率 >= `95%`
- 根因判断人工抽检正确率 >= `80%`
- 修复代码可执行或可通过断言的比例 >= `70%`
- 对简单直接异常题，不能相对 Base 出现大面积回退

这组标准比单纯看“会不会说关键点”更符合本项目真实目标。


## 9. 后续优化方向

结合当前实验结果，后续最值得投入的方向主要有三条：

### 9.1 扩充 benchmark

当前 benchmark 总量只有 `20` 条，统计波动较大。后续建议扩展到 `60-100` 条，并确保每一类错误都有稳定覆盖。

### 9.2 引入可执行评测

为一部分 benchmark 补充：

- `expected_fixed_code`
- 断言脚本
- 期望输出

从而把评测从“文本近似匹配”升级到“修复代码能否真正跑通”。

### 9.3 继续优化 DPO 数据

优先补充以下类型的偏好对：

- `module_not_found / import_error` 这类简单直接错误
- “既要结构合法，又要最小修复”的 chosen/rejected 对
- 信息不足时追问得体、但不过度啰嗦的偏好样本


## 10. 可提炼为简历的项目亮点

下面这些表述可以作为后续简历描述的素材基础：

- 基于 `Qwen3-4B-Instruct-2507` 搭建单文件 Python 代码诊断微调项目，围绕结构化错误分析、最小修复和信息不足追问能力完成 SFT 与 DPO 两阶段训练闭环。
- 自建面向代码报错修复任务的高质量监督数据与偏好数据，覆盖 `TypeError`、`IndexError`、`ImportError`、逻辑错误、上下文不足等多类典型场景。
- 设计并实现 benchmark 推理、离线评估与训练曲线分析流程，对 Base / SFT / DPO 三组模型进行系统对比。
- 在完整 benchmark 上将结构化输出稳定性显著提升到 `100% JSON 合法率`，同时提升错误类型一致性与信息不足场景下的追问准确率。


## 11. 一句话总结

这个项目的核心成果不是“让一个小模型学会回答 Python 报错”，而是把一个本来就有通用能力的 4B 指令模型，进一步收束成了一个更结构化、更诚实、更符合工程约束的代码诊断助手。
